# Upgrade

## Before upgrading

1. **Back up your database** — See [Administration](/self-hosted/administration#backups) for backup instructions
2. **Check the release notes** — Look for breaking changes at [github.com/shellhub-io/shellhub/releases](https://github.com/shellhub-io/shellhub/releases)
3. **Plan for brief downtime** — The upgrade restarts all services, typically under 2 minutes

## Community Edition

### 1. Stop the services

```bash
$ make stop
```

### 2. Pull the latest version

```bash
$ git fetch origin
$ git checkout v<VERSION>
```

Replace `<VERSION>` with the target version tag (e.g., `v0.16.0`).

### 3. Start the services

```bash
$ make start
```

Docker pulls the new container images automatically. Database migrations run on startup.

### 4. Verify the upgrade

```bash
$ ./bin/docker-compose ps
```

All services should show `Up` status.

### Migrating from a pre-v0.24.0 release (MongoDB)

ShellHub used MongoDB before **v0.24.0**, which switched to PostgreSQL as the only supported backend. The MongoDB-to-PostgreSQL migration runs **only on v0.23.x** — there is no in-place migration on v0.24.0 or later.

If your instance still has data in MongoDB, upgrade to a **v0.23.x** release first and let the migration complete (the default `SHELLHUB_DATABASE=migrate` triggers it on startup; watch the API logs until it finishes). Only then upgrade to v0.24.0 or newer, where `SHELLHUB_DATABASE=postgres` is the default. Jumping straight from MongoDB to v0.24.0+ leaves that data behind, so keep your MongoDB backup until you have verified PostgreSQL is working.

## Enterprise Edition

### 1. Authenticate with the Docker registry

Enterprise images are hosted in a private registry. Log in with the credentials provided with your Enterprise license:

```bash
$ docker login registry.infra.ossystems.io
```

### 2. Stop, pull, and start

```bash
$ make stop
$ git fetch origin
$ git checkout v<VERSION>
$ make start
```

> **Note:** If this is your first Enterprise upgrade, make sure `.env.override` includes `SHELLHUB_EDITION=enterprise`.

## Rolling back

If something goes wrong, restore the previous version:

```bash
$ make stop
$ git checkout v<PREVIOUS_VERSION>
$ make start
```

If migrations ran, restore the database backup first — migrations are forward-only.

> **Warning:** Always back up before upgrading. Database migrations are not reversible without a backup.
